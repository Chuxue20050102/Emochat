# 开发问题记录

## 1. App 无法连接后端服务

**问题**：手机 App 运行时报错 `failed to connect to 127.0.0.1:8000`

**原因**：
- 代码硬编码 `127.0.0.1`，手机上指向手机本身而非电脑
- 配置的 IP 地址与电脑实际 IP 不一致

**解决**：
- 创建 `config/index.js` 统一管理 API 地址
- 修改为电脑实际局域网 IP：`172.20.10.3`
- 在 `manifest.json` 添加 H5 开发服务器代理配置
- 确保手机和电脑在同一局域网

---

## 2. AI 聊天始终返回错误提示

**问题**：无论发送什么消息，AI 都返回"抱歉，我刚刚有点走神..."

**原因**：
- Windows 控制台默认 GBK 编码，无法显示 AI 返回的 emoji
- `print()` 抛出 `UnicodeEncodeError`，被 `except` 捕获导致返回错误提示

**解决**：
- 在 `services/ai/utils.py` 添加 `safe_print()` 函数处理编码问题
- 将所有 `print()` 替换为 `safe_print()`

---

## 3. AI 聊天响应慢

**问题**：AI 回复需要较长时间，影响用户体验

**原因**：
- 历史消息上下文过多（默认20条），模型读入成本高
- 单次回复 `max_tokens` 偏大（320），但实际只需要短回复
- Redis 连接每次临时创建，增加请求开销
- 缺少耗时日志，无法定位具体瓶颈

**解决**：
- 在 `services/ai/chat.py` 和 `services/ai/memory.py` 中：
  - 默认上下文条数从 20 降到 8
  - 新增上下文字数上限 `EMOCHAT_MAX_CONTEXT_CHARS=1800`
  - 单次回复 token 上限从 320 降到 180
  - Redis 客户端改为复用，不再每次新建
  - 加了 AI 调用耗时日志：`[AI Chat Timing] ...`
- 在 `chat.py` 中：
  - 加了接口总耗时日志：`[Chat API Timing] ...`

**效果**：
- 回复速度明显提升，尤其是多轮对话后
- 回复更短、更贴近人设设定
- 可通过日志判断慢的具体原因：
  - `AI Chat Timing` 高 → 模型调用慢
  - `Chat API Timing` 远高于 `AI Chat Timing` → 其他链路开销
- 建议：前端显示"AI 正在思考"，后端实现流式输出，长期记忆改为摘要记忆
